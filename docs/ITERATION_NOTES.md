# 迭代记录与复用经验

记录日期：2026-05-08

## 项目定位

这个页面不是“好看的滤镜”，而是一个能在展台现场被游客触屏试玩的显微成像原理演示。核心目标是让玩家边点边理解：荧光图像的颜色不是样本本身的颜色，而是探测器把特定发射光信号翻译成可读伪彩。

## 已确认的体验原则

- 现场使用优先触屏，不依赖鼠标精细操作。
- 打开页面先给明场/焦面感，再进入灰度信号和伪彩显影。
- “手指显影”要保留参与感，但不能让人手速度误导成真实扫描速度。
- 点扫描应表达单点从左到右、逐行采集，而不是从上到下扫整条线。
- 始终只有一个扫描点/扫描头，不能出现多个发射源，也不能像连续发射的激光炮。
- 伪彩要和结构有关系，不能只是把同一张灰度图整体染成红绿蓝。
- 参数文案要解释显微成像概念，不只是说明按钮功能。
- 页面布局要像仪器控制台：左侧样本/输出，中间视野，右侧采集与显示参数。
- Live 视野必须保持正方形，不能因为浏览器宽高变化被拉伸。

## 这轮踩过的坑

- 横向扫描线如果跟随拖拽速度变化，会让人误以为停留时间由手速决定；后来改成扫描读出由程序控制。
- 生成的“细胞/叶绿体/神经纤维”如果没有经典结构特征，会比抽象红绿图更误导；后来改为更明确的核蓝、绿色骨架/叶绿体、红色颗粒/自发荧光。
- 手势互动一旦完全不影响结果，擦亮体验会消失；需要把“局部采集”保留为局部显影，而把“完整点扫描”交给按钮触发。
- 控制面板的术语不能只写 Threshold/Gain/PSF/Offset，必须用科普句子解释它们像什么、影响什么。
- 上传本地图像如果只显示当前图，左侧面板会明显短一截；加入短期缓存后更像仪器里的采集队列。
- 为了三栏底部对齐而强行等高，会让中间正方形视野下方出现大块空洞；最终改成三栏自然高度，避免空白假结构。
- 手机版不能用桌面布局硬压缩；横向样本队列会把页面撑出横向滚动条，最终改成 2x2 小样本格。
- 手机端 `a.download` 可能被 iOS/微信内置浏览器静默拦截；导出图像需要系统分享或预览页兜底。
- 手机端画布不适合按高 DPR 硬渲染；把窄屏 DPR 上限压到 1，可以明显减轻拖拽和扫描时的像素循环负担。
- 样本入口缩略图可以用真实显微/叶脉网图，但要本地化成小资产并记录来源，避免现场网络不稳时入口图消失。
- 上传日常照片不能完全照搬科研样本的通道拆分；照片过曝时绿色通道会吃掉大部分亮区，需要把暗部、结构边缘和高光分别映射到蓝、青绿、红/品红，画面才有展台吸引力。
- 上传照片的 Gain 不能只放大滑块范围，还要同步放宽内部通道限幅；否则界面显示增益提高了，画面亮度却被像素计算截住。
- 手机端调参数需要在视野和控制台之间滚动，点扫描这种会重跑读出的按钮容易被误触；在窄屏触屏下给点扫描加二次确认，桌面端保持一步触发。

## 复用建议

- 大型仪器科普页面可以复用“样本入口 -> 仪器视野 -> 参数控制台”三段式。
- 先把一个准确的核心动作做对，再做视觉修饰；动画不准确会比没有动画更糟。
- 每个参数控件下方都写“概念解释”，不要写“操作说明”。
- 现场交互应该提供明确的一键路径：从默认样本开始，点扫描完整生成，再切伪彩，再微调参数。
- 对移动端单独设计信息密度，不要只缩放桌面版。
- 入口缩略图只负责“辨识与期待”，真实交互样本仍尽量保持代码生成或本地处理，保证离线可用和体验稳定。
- 自定义照片的伪彩要明确是“信号可视化练习”，不是把一张普通照片解释成真实多通道样本；为了现场传播，可以在审美上稍微偏假彩调色。

## 后续可观察问题

- 单位网络打开 GitHub Pages 的速度和稳定性。
- 微信/手机浏览器是否允许本地图像上传和导出。
- 手机导出时系统分享、长按保存、桌面下载三条路径分别是否顺畅。
- iPad 或触屏大屏上拖拽采集是否足够灵敏。
- 人群围观时，是否能在 10 秒内理解“颜色是信号标签”。
- 参数滑块是否需要更大的触摸热区。
- 服务器部署已在手机端验证可登录、可保存、运行顺畅；后续重点可以从“能不能跑”转向“上传照片是否足够好看”。
